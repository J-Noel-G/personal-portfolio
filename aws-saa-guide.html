What is AWS?

Amazon Web Services (AWS) is a comprehensive and broadly adopted cloud computing platform, offering over 200 fully featured services from data centers globally. At its core, AWS provides on-demand computing power (like Amazon EC2), storage (like Amazon S3), databases (like Amazon RDS and DynamoDB), networking, and other foundational services. It allows you to build, deploy, and scale applications quickly and cost-effectively, paying only for what you use instead of buying and maintaining your own physical servers.

Understanding the SAA-C03 Domains

The Solutions Architect - Associate exam is about how to *combine* these services to build secure, resilient, high-performing, and cost-effective solutions. The four domains are the pillars of this "Well-Architected" philosophy and are interconnected.

Domain 1: Design Secure Architectures

This is the foundation. Before you can build anything else, you must know how to secure it. This domain covers identity (who can log in and what can they do, using IAM), network security (creating an isolated network with VPC, firewalls with Security Groups), and data protection (encrypting data at rest and in transit).

Domain 2: Design Resilient Architectures

This is about building applications that can withstand failure. A resilient system doesn't go down just because one server fails. This domain focuses on high availability (using multiple Availability Zones), scalability (using Auto Scaling Groups to handle load), and loose coupling (using services like SQS and SNS so that components can fail independently without breaking the whole system).

Domain 3: Design High-Performing Architectures

Once your application is secure and resilient, you need to make it fast. This domain is about performance optimization. It covers choosing the right compute (e.g., EC2 instance types), storage (e.g., SSD vs. HDD), and database (e.g., RDS vs. DynamoDB) for your specific workload. It also includes caching (with ElastiCache) and global content delivery (with CloudFront).

Domain 4: Design Cost-Optimized Architectures

Finally, you must build all of the above without wasting money. This domain is about financial governance. It covers purchasing options (like Spot Instances and Savings Plans to get discounts), right-sizing (not using an expensive server for a small job), and choosing the most cost-effective storage tiers (like moving old data to S3 Glacier).

Exam Overview

The exam is divided into these four domains, each with a specific weighting:

  • 30% Domain 1: Design Secure Architectures
  • 26% Domain 2: Design Resilient Architectures
  • 24% Domain 3: Design High-Performing Architectures
  • 20% Domain 4: Design Cost-Optimized Architectures

Putting It All Together: A Real-World Example

This guide covers many individual services. The *most important* skill is combining them to solve a business problem. Here is a classic example that touches all four domains.

Scenario: The New E-commerce Website

A startup is launching a new e-commerce store. They expect moderate traffic, but need the site to be 100% available and handle massive, "spiky" traffic spikes during holiday sales. They also must protect customer data.

The "Client" (Business) Perspective:

  • "The site can never go down, especially not during a sale!" (Need: High Availability & Resilience)
  • "If a million people visit on Black Friday, the site must stay fast." (Need: Scalability & Performance)
  • "We are storing customer addresses and payments. We cannot have a data breach." (Need: Security)
  • "I don't want to pay for 20 massive servers all year when I only need them for one weekend." (Need: Cost-Optimization & Elasticity)

The "Expert" (Architect) Solution:

This is the classic "Well-Architected" web application, touching all four domains:

  1. Domain 1 (Security): We start with a VPC, creating Public Subnets for web servers and Private Subnets for the database. Security Groups (firewalls) restrict traffic: the web SG allows port 443 (HTTPS) from the internet, and the database SG allows port 3306 (MySQL) *only* from the web server's SG. We use AWS WAF to block SQL injection and ACM for a free SSL certificate.
  2. Domain 2 (Resilience & Scalability): We use an Auto Scaling Group (ASG) of EC2 instances spanning multiple Availability Zones. An Application Load Balancer (ALB) distributes traffic across them. If an instance or an entire AZ fails, the ALB reroutes traffic. The ASG uses a Target Tracking policy (e.g., "keep CPU at 50%") to automatically launch new instances during the Black Friday spike and terminate them after, achieving elasticity.
  3. Domain 3 (Performance): We use Amazon Aurora (RDS) as our database, which is faster than standard MySQL. For HA, we enable Multi-AZ. To handle read traffic, we add Aurora Read Replicas. We use Amazon ElastiCache (Redis) to cache common database queries in memory. We use Amazon CloudFront (CDN) to cache all our static images and CSS at edge locations worldwide, making the site load instantly for all users.
  4. Domain 4 (Cost-Optimization): The Auto Scaling Group is the primary cost saver, ensuring we only pay for compute we need. Using Aurora Serverless could be an option if traffic is very unpredictable. Caching with ElastiCache and CloudFront also saves money by reducing the load on our backend EC2 instances and database.

Key Services Used: VPC, Security Groups, ALB, EC2 Auto Scaling Group, RDS Aurora (Multi-AZ + Read Replicas), ElastiCache, CloudFront, WAF, ACM.

Key Terms & Core Services

A glossary of the foundational concepts and most common services you must know for the SAA-C03 exam. The four domains of the exam are based on the principles of the Well-Architected Framework.

The 6 Pillars of the AWS Well-Architected Framework

  • 1. Operational Excellence: The ability to run and monitor systems to deliver business value and to continually improve supporting processes and procedures.
  • 2. Security: The ability to protect information, systems, and assets while delivering business value through risk assessments and mitigation strategies. (Exam Domain 1)
  • 3. Reliability: The ability of a workload to perform its intended function correctly and consistently when it’s expected to. This includes the ability to operate and test the workload through its total lifecycle. (Exam Domain 2)
  • 4. Performance Efficiency: The ability to use computing resources efficiently to meet system requirements, and to maintain that efficiency as demand changes and technologies evolve. (Exam Domain 3)
  • 5. Cost Optimization: The ability to run systems to deliver business value at the lowest price point. (Exam Domain 4)
  • 6. Sustainability: The ability to continually improve sustainability impacts by reducing energy consumption and increasing efficiency. (A newer pillar, but good to know).

Core AWS Services by Category

Compute

  • EC2 (Elastic Compute Cloud): Virtual servers in the cloud. The "workhorse" of AWS.
  • Auto Scaling Group (ASG): Automatically adds or removes EC2 instances to meet demand, ensuring performance and fault tolerance.
  • Lambda: Serverless compute. Run code in response to events without managing servers.
  • ECS (Elastic Container Service): AWS's proprietary container orchestration service.
  • EKS (Elastic Kubernetes Service): Managed Kubernetes service for running containerized applications.
  • Fargate: Serverless compute *for containers*. It works with both ECS and EKS, removing the need to manage EC2 instances.

Storage

  • S3 (Simple Storage Service): Infinitely scalable object storage for files, backups, and data lakes. Accessed via API.
  • EBS (Elastic Block Store): A "virtual hard drive" or block-level storage volume that attaches to a *single* EC2 instance.
  • EFS (Elastic File System): A shared, elastic file system for Linux (NFS protocol) that can be mounted by *many* EC2 instances.
  • S3 Glacier: Low-cost, long-term archival storage (e.g., Glacier Deep Archive is the cheapest).
  • Storage Gateway: A hybrid service that connects your on-premises applications to AWS storage.

Database

  • RDS (Relational Database Service): Managed relational databases (e.g., MySQL, PostgreSQL, SQL Server).
  • Aurora: AWS's cloud-native, high-performance database, compatible with MySQL & PostgreSQL. Features Multi-AZ storage.
  • DynamoDB: Fully managed NoSQL key-value database. Delivers single-digit millisecond performance at any scale.
  • ElastiCache: Managed in-memory cache service (Redis or Memcached) to speed up applications.
  • RDS Proxy: A connection pooler for RDS that improves scalability and resilience, especially for Lambda.

Networking & Content Delivery

  • VPC (Virtual Private Cloud): Your own private, isolated network section in AWS where you launch resources.
  • Security Group (SG): A stateful firewall at the *instance* level. Controls inbound/outbound traffic.
  • Network ACL (NACL): A stateless firewall at the *subnet* level. An optional, second layer of defense.
  • Route 53: AWS's managed Domain Name System (DNS) service. Also handles routing policies (failover, latency, etc.).
  • CloudFront: AWS's Content Delivery Network (CDN) that caches content at edge locations globally for low-latency delivery.
  • Application Load Balancer (ALB): Layer 7 (HTTP/S) load balancer. Routes based on content (e.g., URL path).
  • Network Load Balancer (NLB): Layer 4 (TCP/UDP) high-performance load balancer. Can provide a static IP.
  • Direct Connect (DX): A dedicated, private network connection from your on-premises data center to AWS.
  • Transit Gateway (TGW): A central network hub for connecting thousands of VPCs and on-premises networks.

Security, Identity & Compliance

  • IAM (Identity and Access Management): Manages AWS users, groups, roles, and their permissions (policies).
    • * Role: An identity with temporary credentials, assumable by users, services, or external entities.
  • KMS (Key Management Service): Creates and manages encryption keys used to encrypt your data.
  • WAF (Web Application Firewall): Protects web apps (on ALB, CloudFront, API Gateway) from common exploits like SQL injection.
  • Shield: Managed DDoS (Distributed Denial of Service) protection. (Standard is free, Advanced is paid).
  • Secrets Manager: A service for securely storing, retrieving, and automatically rotating secrets (like database passwords).
  • Cognito: Provides user identity management (sign-up/sign-in) for your *applications*, not for AWS.

Management & Governance

  • CloudWatch: The monitoring and observability service. Collects metrics, logs, and sends alarms.
  • CloudTrail: Logs all API calls in your account for auditing, compliance, and governance. "Who did what, when?"
  • CloudFormation: Infrastructure as Code (IaC). Define your entire AWS infrastructure in a JSON or YAML template.
  • Organizations: Centrally manage and govern multiple AWS accounts. Used for consolidated billing and Service Control Policies (SCPs).

Application Integration (Loose Coupling)

  • SQS (Simple Queue Service): A managed message queue service. Used to decouple components by allowing them to send and receive messages asynchronously.
  • SNS (Simple Notification Service): A managed pub/sub (publish/subscribe) service. "Fans out" a single message to many subscribers (e.g., SQS, Lambda, email).
  • EventBridge: A serverless event bus that routes events from various sources (AWS services, custom apps, SaaS) to targets.
  • Step Functions: A serverless visual workflow service for orchestrating multi-step applications (especially Lambda functions).

1.1: Design secure access to AWS resources

This task statement focuses on the foundation of AWS security: controlling who (authentication) can do what (authorization) within your AWS environment. It covers identity management, permissions, and multi-account strategies.

Knowledge of:

Access controls and management across multiple accounts

  • AWS Organizations: A service that allows you to centrally manage and govern your environment as you grow and scale your AWS resources. You can create a "root" organization and invite other accounts to join.
  • Service Control Policies (SCPs): A feature of AWS Organizations. SCPs are "guardrails" that apply to the root, an Organizational Unit (OU), or an individual account. They define the *maximum* permissions allowed for IAM entities in that account. For example, you could use an SCP to block access to services you don't want anyone to use. Important: SCPs do not grant permissions; they only filter or restrict.

AWS federated access and identity services

  • AWS IAM Identity Center (formerly AWS Single Sign-On): The recommended service for managing access across multiple accounts and business applications. It can connect to your external IdP (like Active Directory, Okta, Azure AD) or act as its own IdP. Users log in once and get a portal to access all their assigned AWS accounts and roles.
  • IAM Federation (SAML 2.0 & OIDC): The underlying mechanism. You establish a "trust" relationship between your IdP and AWS. The IdP authenticates the user and sends a SAML assertion to AWS STS, which then provides temporary credentials for an IAM Role.

AWS global infrastructure (for example, Availability Zones, AWS Regions)

Understanding the infrastructure is key to security. A Region is a physical location in the world with multiple, isolated Availability Zones (AZs). An AZ is one or more discrete data centers. IAM is a global service, meaning users, groups, roles, and policies are created globally and are not specific to one Region. However, most other resources (like EC2 instances, VPCs) are Regional.

AWS security best practices (for example, the principle of least privilege)

The Principle of Least Privilege: This is the most critical concept. You should grant only the minimum permissions necessary for a user, group, or service to perform its required tasks, and nothing more. Avoid using overly permissive policies like "Effect": "Allow", "Action": "*", "Resource": "*".

The AWS shared responsibility model

AWS is responsible for the security OF the cloud (hardware, global infrastructure, managed services). You, the customer, are responsible for security IN the cloud (your data, IAM configuration, OS patching, network security groups, encryption). For example, with EC2, AWS secures the data center, but you must secure the operating system and the data on the instance.

Skills in:

Applying AWS security best practices to IAM users and root users

  • Root User: This is the "owner" account. It has unrestricted access. Best Practice: Do NOT use the root user for daily tasks.
  • MFA (Multi-Factor Authentication): You MUST secure the root user with MFA. You should also enforce MFA for all IAM users, especially administrative ones.
  • Access Keys: Do not generate long-term access keys (AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY) for the root user. If you must use them for programmatic access, create an IAM user or role instead.

Designing a flexible authorization model that includes IAM users, groups, roles, and policies

  • Users: An entity (person or application) that interacts with AWS.
  • Groups: A collection of IAM users. You apply policies to a group, and all users in that group inherit those permissions. This is much easier than managing policies for individual users.
  • Roles: An identity with permissions that can be "assumed" by a trusted entity. Roles do not have their own long-term credentials. They are the preferred way to grant permissions to services (e.g., an EC2 instance needing to write to S3) or for federated/cross-account access.
  • Policies: The JSON documents that define permissions. An Identity-based Policy is attached to a user, group, or role. A Resource-based Policy is attached to a resource (e.g., an S3 bucket policy).
Exam Tip: An Identity-based Policy (on an IAM user) answers "What can this user do?". A Resource-based Policy (on an S3 bucket) answers "Who can access this resource?". Both are evaluated together. A Deny in *either* policy will always win.

Designing a role-based access control strategy (for example, AWS STS, role switching, cross-account access)

To allow an IAM user in Account A (Dev) to access resources in Account B (Prod), you would:

  1. In Account B (Prod), create an IAM Role (e.g., ProdReadOnlyRole) that has the required permissions (e.g., s3:GetObject).
  2. In that role's "Trust Policy," specify that it trusts Account A.
  3. In Account A (Dev), grant your IAM user (e.g., DevUser) permission to call sts:AssumeRole on the ProdReadOnlyRole ARN.
  4. The DevUser can then "switch roles" (which uses AWS STS) to get temporary credentials for Account B.

Designing a security strategy for multiple AWS accounts (for example, AWS Control Tower, service control policies [SCPs])

AWS Control Tower provides the easiest way to set up and govern a new, secure, multi-account AWS environment based on best practices. It automates the setup of AWS Organizations, creates core OUs (like "Security" and "Sandbox"), and implements preventive and detective "guardrails" (which are based on SCPs and AWS Config rules).

Determining the appropriate use of resource policies for AWS services

Resource-based policies are powerful for cross-account access and public access. The most common example is an S3 Bucket Policy. You can use it to:

  • Grant public read access to objects.
  • Grant another AWS account (or specific users in it) access to your bucket.
  • Deny access if the request doesn't come from a specific IP address or VPC.

Determining when to federate a directory service with IAM roles

You should federate when you already have an existing identity provider (like Microsoft Active Directory, Okta, Ping, or Azure AD) and you want your users to use their existing corporate credentials to access AWS. This avoids managing two sets of passwords, simplifies on-boarding/off-boarding, and centralizes identity management.

1.2: Design secure workloads and applications

This section moves from "who" can log in to "how" your applications and network are protected from threats. This is primarily about VPC design, network firewalls, and application-layer security.

Knowledge of:

Application configuration and credentials security

Never hardcode secrets! Do not store database passwords, API keys, or other credentials in your code or EC2 User Data. Use a dedicated service:

  • AWS Secrets Manager: The preferred service for secrets. It can store, retrieve, and automatically rotate credentials (e.g., RDS database passwords). Applications can query the Secrets Manager API (using an IAM role) to get credentials at runtime.
  • AWS Systems Manager (SSM) Parameter Store: Can store plain-text or encrypted (using KMS) configuration data and secrets. It's a good general-purpose service. Secrets Manager is better for automated rotation.
Exam Tip: Choose Secrets Manager if you need automatic key rotation (e.g., for RDS). Choose SSM Parameter Store for simpler configuration data or secrets that don't need rotation. Parameter Store has a free standard tier, while Secrets Manager has a cost per secret.

AWS service endpoints

By default, when your EC2 instance communicates with a service like S3, the traffic goes over the public internet. VPC Endpoints allow you to keep this traffic within the AWS network, improving security and performance.

  • Gateway Endpoints: A gateway that you add as a target in your route table. It's free. Only supports S3 and DynamoDB.
  • Interface Endpoints (AWS PrivateLink): Uses an Elastic Network Interface (ENI) with a private IP address in your subnet. It acts as an entry point to the service. It supports most AWS services, costs money, and works with on-premises connections (via VPN or Direct Connect).

Control ports, protocols, and network traffic on AWS

  • Security Groups (SGs): Act as a firewall at the *instance level*. They are stateful (if you allow inbound port 80, the outbound return traffic is automatically allowed). You can only create allow rules.
  • Network ACLs (NACLs): Act as a firewall at the *subnet level*. They are stateless (if you allow inbound port 80, you must *also* explicitly allow outbound ports 32768-65535 for the return traffic). They have allow and deny rules, processed in order.
Exam Tip: Remember SGs are stateful/instance-level, and NACLs are stateless/subnet-level. SGs are your first line of defense; NACLs are a broader, secondary defense.

Secure application access

Amazon Cognito: This service is for your *application's users*, not your AWS account users (that's IAM). It provides authentication, authorization, and user management for web and mobile apps.

  • User Pools: A user directory. Handles user sign-up, sign-in, and password reset. Can federate with social (Google, Facebook) and SAML IdPs.
  • Identity Pools: Grants users (from a User Pool or other IdP) temporary AWS credentials to access specific AWS services (e.g., allows an app user to upload directly to an S3 bucket).

Security services with appropriate use cases

  • Amazon GuardDuty: A smart, "hands-off" threat detection service. It continuously monitors your AWS account's CloudTrail logs, VPC Flow Logs, and DNS logs for malicious activity (e.g., crypto-mining, communication with known bad IPs, port scanning) and alerts you.
  • Amazon Macie: A data security service that uses machine learning to discover, classify, and protect sensitive data (like PII or financial info) stored in Amazon S3.

Threat vectors external to AWS

  • DDoS (Distributed Denial of Service): An attempt to overwhelm an application with traffic.
    • AWS Shield Standard: Free, automatically enabled for all customers. Protects against common Layer 3/4 DDoS attacks.
    • AWS Shield Advanced: A paid service for enhanced protection against large-scale DDoS attacks. Includes 24/7 DDoS response team support and integrates with AWS WAF.
  • SQL Injection / Cross-Site Scripting (XSS): Application-layer attacks.
    • AWS WAF (Web Application Firewall): A Layer 7 firewall that lets you block common web exploits like SQL injection and XSS. You can apply it to an Application Load Balancer (ALB), API Gateway, or CloudFront.

Skills in:

Designing VPC architectures with security components

A standard, secure VPC design is the "multi-tier" architecture:

  • VPC: A logically isolated network.
  • Subnets: A range of IP addresses in your VPC.
    • Public Subnets: Have a route table entry pointing to an Internet Gateway (IGW). Resources here (like web servers or NAT Gateways) can be reached from the internet.
    • Private Subnets: Do not have a route to an IGW. Resources here (like application servers or databases) are isolated.
  • NAT Gateway (Network Address Translation): A managed AWS service that you place in a *public subnet*. It allows instances in a *private subnet* to initiate outbound traffic to the internet (e.g., for software updates) while blocking all inbound traffic.
  • Security Groups & NACLs: Used to control traffic flow. E.g., the web-tier SG allows port 443 from 0.0.0.0/0. The app-tier SG allows port 8080 only from the *web-tier SG*. The db-tier SG allows port 3306 only from the *app-tier SG*.

Determining network segmentation strategies

This is the public/private subnet strategy described above. By segmenting your network, you create a "defense-in-depth" model. If an attacker compromises your web server (public), they are still firewalled off from your database (private).

Integrating AWS services to secure applications

A secure architecture combines services. A common pattern is:

Route 53 (DNS) -> CloudFront (CDN, with Shield Advanced) -> AWS WAF (blocking SQLi) -> Application Load Balancer (distributing traffic) -> EC2 Instances (in a private subnet) -> Secrets Manager (to get DB creds) -> RDS Database (in a private subnet).

This architecture uses AWS Shield, WAF, and Secrets Manager to secure the application at multiple layers.

Securing external network connections to and from the AWS Cloud

When you need to connect your on-premises data center to your VPC:

  • AWS Site-to-Site VPN: Creates an encrypted (IPsec) tunnel between your VPC (using a Virtual Private Gateway) and your on-premises network (using a Customer Gateway). This connection travels over the public internet.
  • AWS Direct Connect (DX): A dedicated, private, physical network connection between your data center and an AWS DX location. It's high-bandwidth, consistent, and does not use the internet. It's more expensive and takes longer to set up.

1.3: Determine appropriate data security controls

This section is all about protecting your data itself, primarily through encryption (at rest and in transit) and controlling access to that data.

Knowledge of:

Data access and governance

This involves using policies to control who can access what data. This includes IAM policies, S3 Bucket Policies, and S3 Access Points. Governance involves auditing access (CloudTrail) and managing data lifecycles.

Data recovery

This means having a plan to recover data in case of accidental deletion or corruption. Key services are:

  • AWS Backup: A centralized, managed service to automate backups across services like EBS, RDS, EFS, and DynamoDB.
  • S3 Versioning: Keeps a full copy of every version of an object. Allows you to "roll back" to a previous version or recover a deleted object.
  • EBS & RDS Snapshots: Point-in-time backups of your volumes and databases.

Data retention and classification

  • Data Retention: How long you need to keep data. This is often for compliance.
    • S3 Lifecycle Policies: Automate the movement of data to cheaper storage tiers (e.g., Standard -> Glacier) and its eventual deletion.
    • S3 Object Lock: Enforces WORM (Write-Once, Read-Many) policies. Used for compliance to ensure data cannot be deleted or overwritten for a fixed period.
  • Data Classification: Identifying what data is sensitive (e.g., PII). Amazon Macie is the key service here for S3.

Encryption and appropriate key management

This is the most important topic in this section. AWS Key Management Service (KMS) is the central service for creating and managing encryption keys.

Skills in:

Aligning AWS technologies to meet compliance requirements

If you need to meet compliance standards like PCI-DSS or HIPAA:

  • AWS Artifact: A self-service portal where you can download AWS's compliance reports (e.g., SOC 2 report) to show to your auditors.
  • AWS CloudTrail: Records every API call in your account. This provides the audit trail (who did what, when) required for compliance.
  • AWS Config: Assesses, audits, and evaluates the configurations of your resources. You can create rules (e.g., "all EBS volumes must be encrypted") and be alerted if a resource becomes non-compliant.

Encrypting data at rest (for example, AWS Key Management Service [AWS KMS])

This means encrypting data that is stored on disk. Most AWS storage services offer encryption, typically using keys from KMS.

  • Server-Side Encryption (SSE): AWS encrypts the data for you after receiving it.
    • SSE-S3: S3 manages the encryption keys. Easiest, no extra cost.
    • SSE-KMS: S3 calls KMS to use a key that you manage (either AWS-managed or Customer-Managed). This gives you control over the key and an audit trail in CloudTrail.
    • SSE-C: You provide your own encryption key with every request. AWS does not store the key.
    • Exam Tip: Use SSE-S3 for basic, "check-the-box" encryption. Use SSE-KMS when you need to control the key, manage its rotation, or provide an audit trail of who used the key (via CloudTrail). SSE-C is rare and used when you have strict requirements to manage your own keys outside of AWS.
  • EBS Encryption: You can enable a setting to encrypt all new EBS volumes by default. This uses KMS.
  • RDS Encryption: You can enable encryption when creating a database. This also uses KMS.

Encrypting data in transit (for example, AWS Certificate Manager [ACM] using TLS)

This means encrypting data as it travels over a network (e.g., HTTPS).

  • AWS Certificate Manager (ACM): Provides free, auto-renewing public SSL/TLS certificates. You can easily attach these to Application Load Balancers (ALBs) and CloudFront Distributions to enable HTTPS.
  • Enforce Encryption: You should configure your ALB or CloudFront to redirect all HTTP traffic to HTTPS.

Implementing access policies for encryption keys

In KMS, the Key Policy is the primary document that controls access. It defines who can *manage* the key and who can *use* the key (for encrypt/decrypt operations). You must attach a key policy. IAM policies can also be used to grant permissions, but only if the key policy allows it.

Implementing data backups and replications

  • Backups: Use AWS Backup, or manually create EBS/RDS snapshots.
  • Replication:
    • S3 Cross-Region Replication (CRR): Automatically copies objects from a bucket in one region to a bucket in another. Used for DR and lower latency.
    • RDS Read Replicas: Can be created in the same region or a different region (for DR).

Implementing policies for data access, lifecycle, and protection

This combines several skills. For example, a complete S3 strategy would include:

  1. A Bucket Policy to restrict access (e.g., deny deletes, require encryption).
  2. Versioning and MFA Delete enabled for protection.
  3. A Lifecycle Policy to move 30-day-old data to S3-IA and 90-day-old data to Glacier Deep Archive.

Rotating encryption keys and renewing certificates

  • KMS Key Rotation: For Customer Managed Keys (CMKs), you can enable automatic annual key rotation. KMS handles this for you.
  • Certificate Renewal: Certificates provisioned from ACM that are attached to AWS services (like ALB, CloudFront) will be automatically renewed by AWS.

2.1: Design scalable and loosely coupled architectures

This is one of the most important domains. It's about building modern applications that can handle growth (scalability) and where components are independent (loose coupling), which improves fault tolerance and maintainability.

Knowledge of:

API creation and management

Amazon API Gateway: A fully managed service that acts as the "front door" for your application's backend. It's often used with AWS Lambda. It can handle API versioning, request/response transformation, user authentication (e.g., with Cognito), request throttling, and caching.

AWS managed services with appropriate use cases

The core of loose coupling is using messaging services to decouple your components:

  • Amazon SQS (Simple Queue Service): A fully managed message queue. A component (producer) sends a message to a queue. Another component (consumer) polls the queue, retrieves the message, processes it, and then deletes it. This decouples the producer and consumer. If the consumer fails, messages simply pile up in the queue, and no data is lost.
    • Standard Queue: At-least-once delivery, best-effort ordering. High throughput.
    • FIFO Queue: Exactly-once processing, first-in-first-out ordering. Lower throughput.
  • Amazon SNS (Simple Notification Service): A fully managed pub/sub (publish/subscribe) service. A producer sends one message to an SNS "topic." SNS then "fans out" that message to all subscribers (e.g., SQS queues, Lambda functions, email addresses, SMS).
  • Other managed services like AWS Transfer Family (for SFTP) and Secrets Manager (for credentials) also reduce operational overhead and are inherently resilient.
Exam Tip: Use SQS for decoupling two components (one-to-one). Use SNS for fan-out (one-to-many). They are often used together: SNS -> SQS.

Caching strategies

Caching is used to store frequently accessed data in a faster, temporary location to reduce latency and load on your backend (like a database).

  • Amazon ElastiCache: A managed in-memory cache service. Supports Redis (more features: data persistence, replication) and Memcached (simpler). Used to cache database query results or user session data.
  • Amazon CloudFront (CDN): A global Content Delivery Network. It caches your static and dynamic content at "Edge Locations" around the world, closer to your users. This dramatically reduces latency.
  • API Gateway Caching: Can cache responses from your API, reducing the number of calls to your backend (e.g., Lambda).

Design principles for microservices

Microservices architecture breaks a large application into a collection of smaller, independent services. A key principle is making these services stateless. A stateless workload (like a Lambda function or a container) does not store any persistent data on its local disk. All "state" (like a user's session) must be externalized to a database (like DynamoDB) or a cache (like ElastiCache).

Event-driven architectures

An architecture where services react to "events." For example: a user uploads an image to S3 (the event), which triggers a Lambda function (the compute) to create a thumbnail. Amazon EventBridge is a serverless event bus that simplifies building event-driven apps by routing events from AWS services, custom apps, and SaaS partners to targets like Lambda.

Exam Tip: SQS vs. SNS vs. EventBridge
  • SQS: One-to-one, asynchronous decoupling. A message is processed by *one* consumer. Use it to buffer or queue work.
  • SNS: One-to-many, "fan-out." A message is sent to *all* subscribers. Use it for notifications or to trigger multiple independent actions.
  • EventBridge: Many-to-many, event-based routing. It's like SNS but much more powerful. It can *filter* events based on their content (the event "payload") and route them to different targets. It's the standard for building modern event-driven applications.

Horizontal scaling and vertical scaling

  • Vertical Scaling: Increasing the size of a single instance (e.g., moving from a t3.micro to a t3.large). You eventually hit a limit, and it usually requires downtime.
  • Horizontal Scaling (Scaling Out): Adding *more* instances. This is the cloud-native, preferred way to scale. It's an almost "infinite" capacity and can be done automatically with Auto Scaling Groups (ASGs).

Load balancing concepts

A load balancer distributes incoming traffic across multiple targets (like EC2 instances) in multiple Availability Zones. This is essential for both scalability and high availability.

  • Application Load Balancer (ALB): Layer 7 (HTTP/S). "Content-based" routing. It can route traffic based on the URL path (e.g., /api goes to one set of servers, /images goes to another). Ideal for microservices.
  • Network Load Balancer (NLB): Layer 4 (TCP/UDP). "Flow-based" routing. It's extremely fast, has ultra-low latency, and can provide a static IP address per AZ. Ideal for high-performance or non-HTTP traffic.
  • Gateway Load Balancer (GWLB): Layer 3/4. Used to route traffic through third-party virtual appliances (e.g., firewalls, packet inspection).

Multi-tier architectures

The classic "Web Tier, App Tier, DB Tier" design, typically segmented using public and private subnets. (Covered in 1.2).

Serverless technologies and patterns

  • AWS Lambda: Function-as-a-Service (FaaS). You upload your code, and it runs in response to an event (like an API call or S3 upload). You pay only for the milliseconds it runs.
  • AWS Fargate: Serverless compute *for containers* (used with ECS or EKS). You just define your container and its resource needs, and Fargate runs it without you ever seeing the underlying EC2 instance.
  • Other serverless services: S3, DynamoDB, API Gateway, SQS, SNS, EventBridge.

Storage types with associated characteristics

  • Object (Amazon S3): Infinitely scalable, highly durable, accessed via HTTP API. For files, backups, data lakes, static websites.
  • File (Amazon EFS): A managed Network File System (NFS) that can be mounted by *many* EC2 instances at once. Scales automatically.
  • Block (Amazon EBS): A "virtual hard drive" that attaches to a *single* EC2 instance. It's what your EC2 instance boots from.
Exam Tip: S3 = "Infinite", shared via web. EFS = Shared file system for Linux EC2. EBS = Hard drive for one EC2.

Orchestration of containers

  • Amazon ECS (Elastic Container Service): AWS's proprietary, simple container orchestrator.
  • Amazon EKS (Elastic Kubernetes Service): AWS's managed Kubernetes service. More complex, but industry-standard.
  • Both can run on EC2 instances you manage or on Fargate (serverless).

When to use read replicas

In a database, a Read Replica is a read-only copy of your primary database. You use them to offload read-heavy traffic from your primary DB, improving read performance. They use *asynchronous* replication.

Workflow orchestration

AWS Step Functions: A serverless visual workflow service. It's used to coordinate multiple AWS services (especially Lambda functions) into a complete, stateful workflow. It handles retries, error handling, and parallel execution.

Skills in:

Designing event-driven, microservice, and/or multi-tier architectures

This is about combining the services above.

  • Serverless Microservice: API Gateway (API) -> Lambda (compute) -> DynamoDB (database).
  • Event-Driven: S3 object upload (event) -> SNS (fan-out) -> SQS (buffer) -> Lambda (process).
  • Multi-Tier: ALB (load balancer) -> EC2 ASG (web tier) -> EC2 ASG (app tier) -> RDS (database).

Determining scaling strategies for components

An Auto Scaling Group (ASG) scales EC2 instances. Key strategies:

  • Dynamic Scaling: "Scale out when CPU > 70%."
  • Target Tracking: A type of dynamic scaling. "Keep the average CPU at 50%." (This is the simplest and preferred method).
  • Scheduled Scaling: "Scale out to 10 instances at 9 AM, scale in to 2 at 5 PM."

Determining the AWS services required to achieve loose coupling

The answer is almost always Amazon SQS, Amazon SNS, or Amazon EventBridge. These services break the direct dependencies between components.

Determining when to use containers vs. serverless

  • Use Serverless (Lambda) for: Event-driven tasks, short-running jobs (max 15 min), APIs that have inconsistent traffic (spiky).
  • Use Containers (ECS/EKS with Fargate/EC2) for: Migrating existing applications, long-running processes, complex microservices, or when you need fine-grained control over the compute environment (e.g., specific GPU).

Recommending and using purpose-built AWS services

This skill involves choosing the right tool for the job. For example, instead of running a MySQL database on EC2 (which requires manual management), use RDS. Instead of building a message queue with RabbitMQ, use SQS. Instead of writing custom code to analyze text, use Amazon Comprehend. Using managed, purpose-built services improves resilience and reduces operational overhead.

Real-World Example: Serverless Thumbnail Generator

Scenario: The Serverless Photo Thumbnail Generator

A mobile app startup wants users to upload photos. Every time a user uploads a "profile.jpg" to a specific S3 bucket, a 100x100 thumbnail must be created and saved in a *different* bucket.

The "Client" (Business) Perspective:

  • "It needs to happen instantly when the user uploads." (Need: Event-Driven, Low Latency)
  • "We might get 10 uploads one day and 10 million the next. I have no idea." (Need: Massive & Automatic Scalability)
  • "This is a tiny feature. I refuse to pay for a 24/7 server just to wait for uploads." (Need: Extreme Cost-Optimization / Pay-per-use)

The "Expert" (Architect) Solution:

This is a classic serverless, event-driven pattern that perfectly demonstrates loose coupling.

  1. Storage: We create two S3 Buckets: app-uploads and app-thumbnails.
  2. The Event: We configure an S3 Event Notification on the app-uploads bucket. We set it to trigger on s3:ObjectCreated:Put.
  3. The Compute: The target for that event will be an AWS Lambda function.
  4. The Logic: We write a simple Lambda function that receives the event, gets the image from app-uploads, resizes it, and puts the new thumbnail into the app-thumbnails bucket.
  5. Security: The Lambda function is given an IAM Role that grants it *only* the permissions it needs: s3:GetObject on app-uploads/* and s3:PutObject on app-thumbnails/* (Principle of Least Privilege).

Why this is the perfect solution:

  • Cost (Domain 4): If there are 0 uploads, you pay **$0.00**. You are *only* charged for the milliseconds the Lambda function runs.
  • Scalability (Domain 2): If 10,000 users upload at the *exact same second*, AWS will simply run 10,000 concurrent copies of your Lambda function. The scaling is massive and automatic.
  • Resilience (Domain 2): It's fully "loosely coupled." S3 and Lambda are both highly available managed services. There are no servers to manage or patch.

Key Services Used: S3, AWS Lambda, IAM.

2.2: Design highly available and/or fault-tolerant architectures

High Availability (HA) means your application can withstand the failure of one or more components (like an EC2 instance or an entire data center) without downtime. Fault Tolerance is the ability to withstand failure. They are closely related.

Knowledge of:

AWS global infrastructure (for example, Availability Zones, AWS Regions)

This is the foundation of HA.

  • Region: A geographically isolated area (e.g., us-east-1).
  • Availability Zone (AZ): One or more discrete data centers within a Region, with redundant power, networking, and connectivity. AZs are physically separate.

Rule #1 of HA: To be highly available, you must run your application across multiple Availability Zones. A failure of a single AZ should not take your application offline.

AWS managed services and distributed design patterns

Using AWS managed services (like S3, RDS, DynamoDB, or even AI services like Amazon Comprehend or Amazon Polly) inherently improves resilience, as AWS manages the high availability of the service itself. This is part of a distributed design, where you rely on managed, resilient components instead of building them yourself.

Disaster recovery (DR) strategies

DR is about recovering from a large-scale disaster (like an entire Region failing). This is different from HA (which is usually about AZ failure). You must know the DR strategies and the two key metrics:

  • RPO (Recovery Point Objective): How much data can you afford to *lose*? (Measured in time, e.g., 15 minutes of data loss).
  • RTO (Recovery Time Objective): How *long* can your application be down? (Measured in time, e.g., 1 hour of downtime).

The four DR strategies, from cheapest/slowest to most expensive/fastest:

  1. Backup and Restore: Back up your data (e.g., S3, RDS Snapshots) to your DR region. To recover, you deploy your infrastructure (from CloudFormation) and restore the data. Highest RTO/RPO.
  2. Pilot Light: Your core infrastructure (e.g., your database) is always running in the DR region, but scaled down. App servers are off. To recover, you turn on/scale up the app servers and point DNS. Faster RTO/RPO.
  3. Warm Standby: A scaled-down, but fully functional, version of your application is always running in the DR region. To recover, you scale it up and point DNS. Low RTO/RPO.
  4. Multi-Site Active-Active: A full-scale application is running in both regions. Amazon Route 53 distributes traffic to both. If one region fails, Route 53 routes all traffic to the healthy one. Near-zero RTO/RPO. Most complex and expensive.

Failover strategies

Amazon Route 53 is key for DNS-level failover, especially for DR. You can configure a Failover Routing Policy. Route 53 monitors the health of your primary region's endpoint. If it becomes unhealthy, Route 53 automatically starts routing all traffic to your secondary (DR) endpoint.

Immutable infrastructure

This is a principle where you *never* log in to a running server to patch or update it. Instead, to deploy a new version, you build a new, patched machine image (AMI), deploy a new set of servers from that image, test them, and then terminate the old servers. This is often called a Blue/Green deployment. It's safer and more repeatable.

Load balancing concepts (for example, Application Load Balancer)

ALBs and NLBs are a critical HA component. They are designed to be highly available themselves and will automatically route traffic *only* to the healthy instances in your Auto Scaling Group. If an instance in one AZ fails a health check, the ALB stops sending it traffic.

Proxy concepts (for example, Amazon RDS Proxy)

RDS Proxy: A managed database connection pooler. Applications (like Lambda) often open and close database connections very rapidly, which can overwhelm a database. RDS Proxy sits in front of your RDS database, pools and shares connections, and makes applications more resilient to database failovers (it holds app connections open while the DB fails over).

Service quotas and throttling

Every AWS account has service quotas (formerly "limits") (e.g., you can only run 20 On-Demand EC2 instances). AWS services also "throttle" API calls to prevent abuse. For a resilient architecture, you must monitor your usage against quotas and request increases *before* you hit them. For DR, you must ensure your quotas in your standby region are high enough to run your full production load.

Storage options and characteristics (for example, durability, replication)

  • S3: 99.999999999% (11 9s) of durability. It automatically stores your data across at least 3 AZs. It's natively HA and durable.
  • EBS: A block store tied to a *single AZ*. If the AZ fails, the EBS volume is unavailable. To make it durable, you must take EBS Snapshots (which are stored in S3).
  • EFS: A file system that stores data across *multiple AZs* within a region. It's natively HA.
  • RDS Multi-AZ: This is a key HA feature. When enabled, RDS creates a *synchronous* standby replica of your database in a *different AZ*. If your primary DB fails, RDS automatically fails over to the standby with no data loss (RPO=0) and minimal downtime (RTO=minutes). This is for HA, *not* for read scaling (that's Read Replicas).

Workload visibility (for example, AWS X-Ray)

To build a resilient system, you must know when it's failing. AWS X-Ray is a service that helps you debug and analyze microservices and distributed applications. It traces requests as they travel through your application (e.g., from API Gateway -> Lambda -> DynamoDB), showing you a service map and identifying bottlenecks or errors.

Skills in:

Determining automation strategies to ensure infrastructure integrity

AWS CloudFormation: Infrastructure as Code (IaC). You define your entire infrastructure (VPC, EC2, SGs, etc.) in a JSON or YAML template. This allows you to deploy your *exact* infrastructure stack repeatably, in any region. This is essential for DR automation.

Determining the AWS services required to provide a highly available and/or fault-tolerant architecture

A typical HA architecture includes:

  • Route 53 for DNS.
  • Application Load Balancer distributing traffic across...
  • Multiple AZs. Inside each AZ, you have...
  • An Auto Scaling Group managing EC2 instances.
  • An RDS Multi-AZ Database for state.
  • This design requires identifying the right metrics for health checks (e.g., HTTP 200) and scaling (e.g., CPU) to deliver on business requirements (e.g., "99.9% uptime").

Implementing designs to mitigate single points of failure

A single point of failure (SPOF) is any component that would take down your app if it failed.

  • SPOF: An EC2 instance in one AZ. Fix: Use an ASG with instances in multiple AZs.
  • SPOF: An RDS database in one AZ. Fix: Enable RDS Multi-AZ.
  • SPOF: A NAT Gateway in one AZ. Fix: Deploy a NAT Gateway in *each* AZ that contains private subnets, and configure route tables accordingly. This also involves designing resilient route tables that can fail over.

Exam Tip: The NAT Gateway is a classic Single Point of Failure (SPOF). If the AZ it's in fails, all private instances in *other* AZs that route through it will lose internet access. The resilient solution is to deploy a NAT Gateway in *each* AZ and have instances in that AZ use their local gateway.

Implementing strategies to ensure the durability and availability of data

  • Durability (Data won't be lost): S3 (by default), EBS Snapshots, RDS Snapshots, S3 Versioning.
  • Availability (Data can be accessed): S3 (by default), EFS (by default), RDS Multi-AZ.

Selecting an appropriate DR strategy to meet business requirements

This is a classic exam scenario. They will give you an RTO and RPO, and you must pick the DR strategy (Backup/Restore, Pilot Light, Warm Standby, Active-Active) that meets those requirements at the lowest cost.

Using AWS services to improve the reliability of legacy applications

If you have a legacy application that you cannot change, you can still improve its reliability. For example, if an application constantly opens/closes database connections, placing RDS Proxy in front of the database can absorb this "bad" behavior and protect the database, improving reliability without any application code changes.

3.1: Determine high-performing and/or scalable storage solutions

This section focuses on selecting the right storage service based on its performance (IOPS, throughput) and scalability characteristics.

Knowledge of:

Storage types with associated characteristics (object, file, block)

This is the fundamental concept for all storage decisions.

  • Object (Amazon S3): Accessed via HTTP API (GET, PUT, POST). Not mountable as a file system. Delivers massive scalability (unlimited storage) and high throughput. Ideal for web content, backups, data lakes, and static assets.
  • File (Amazon EFS): A shared file system (NFS protocol) that can be mounted by thousands of EC2 instances simultaneously (Linux-based). It's elastic, growing and shrinking as you add/remove files. Performance is measured in throughput.
  • Block (Amazon EBS): A "virtual hard drive" that attaches to a *single* EC2 instance in the *same AZ*. Performance is measured in IOPS (Input/Output Operations Per Second) and Throughput (MB/s). This is your boot volume and is used for high-performance databases or applications requiring low-latency disk access.

Storage services with appropriate use cases

  • Amazon S3: Use for "unlimited" storage, high-throughput data lakes, website assets, and backups.
    • S3 Standard: Highest performance, for frequently accessed data.
    • S3 Intelligent-Tiering: Automatically moves data between frequent and infrequent access tiers to optimize costs, with no performance impact.
  • Amazon EFS: Use when you need a *shared file system* for multiple EC2 instances. Examples: a shared code repository, a content management system's web root.
    • Performance Modes: General Purpose (default, low latency) vs. Max I/O (higher throughput, higher latency, for large-scale data processing).
    • Throughput Modes: Bursting (default, throughput scales with file system size) vs. Provisioned (you set a fixed throughput).
    • Exam Tip: For EFS, use General Purpose performance mode for most workloads (like web servers). Only use Max I/O for highly parallelized, high-throughput jobs where latency doesn't matter. For throughput, use Bursting unless you have a large file system that needs more throughput than its size provides, then use Provisioned.
  • Amazon EBS: Use for boot volumes and for any single-instance, high-performance workload (like a relational database).
    • General Purpose SSD (gp3, gp2): The default. A balance of price and performance. gp3 is the modern standard, allowing you to provision IOPS and throughput independently of size.
    • Provisioned IOPS SSD (io2 Block Express, io1): Highest performance. For critical, I/O-intensive workloads (e.g., large production databases) where you need to guarantee a high number of IOPS (up to 256,000 with io2 Block Express).

Hybrid storage solutions to meet business requirements

AWS Storage Gateway: A service that connects your on-premises environment to AWS cloud storage.

  • S3 File Gateway: Presents an S3 bucket as an NFS or SMB file share to your on-premises servers. A "front-end" cache provides low-latency access to frequently used files.
  • FSx File Gateway: Presents your Amazon FSx for Windows File Server as an on-premises SMB share, providing a fully managed Windows file server backend with local caching.
  • Volume Gateway: Presents cloud storage as iSCSI block volumes to your on-premises applications.
  • Tape Gateway: Replaces physical tapes with virtual tapes backed by S3 and Glacier.

Skills in:

Determining storage services and configurations that meet performance demands

This is about matching the workload to the service:

  • Workload: Big data analytics, massive parallel reads. Solution: S3. It has massive parallel throughput.
  • Workload: Shared web content for a fleet of EC2 instances. Solution: EFS.
  • Workload: A high-performance production database (MySQL, PostgreSQL). Solution: EBS with Provisioned IOPS (io2) volumes attached to an EC2 instance.
  • Workload: A read-heavy application on a single EC2 instance. Solution: Use an EBS volume with an EC2 Instance Store (fast, *ephemeral* block storage) as a local cache.

Determining storage services that can scale to accommodate future needs

  • S3: Scales "infinitely" by default.
  • EFS: Scales elastically (storage and throughput) by default.
  • EBS: Does *not* scale automatically. You must manually resize an EBS volume (which can be done while in-use, but may require OS-level commands) or rely on EC2 Auto Scaling, which scales by adding *new instances* with *new volumes*.

3.2: Design high-performing and elastic compute solutions

This section is about choosing the right compute service and size, and ensuring it can scale to meet demand.

Knowledge of:

AWS compute services with appropriate use cases

  • Amazon EC2: Virtual servers. The "IAAS" (Infrastructure-as-a-Service) workhorse. You manage the OS and everything on it.
  • AWS Lambda: Serverless functions. For short-running, event-driven tasks. Scales automatically from 0 to thousands of concurrent executions.
  • AWS Fargate: Serverless compute for containers (ECS/EKS). You don't manage the underlying EC2 instances.
  • Amazon ECS/EKS on EC2: You manage the cluster of EC2 "worker nodes" that your containers run on. This gives you more control (e.c., for GPUs) but requires more management.
  • AWS Batch: For running batch computing jobs (e.g., scientific computing, financial analysis) at scale. It manages a queue of jobs and provisions the right compute (EC2 or Spot) to run them.
  • Amazon EMR (Elastic MapReduce): Managed big data platform (e.g., Spark, Hadoop) for processing and analyzing massive datasets.

Distributed computing concepts

This refers to using AWS's global infrastructure. For a global application, you can deploy your compute (EC2, Lambda) in multiple Regions closer to your users. You can also use edge services like Lambda@Edge (which runs your Lambda function at CloudFront edge locations) for ultra-low latency request processing.

Queuing and messaging concepts

As covered in 2.1, SQS is key for performance. It allows a front-end (e.g., an API) to accept requests and "queue" them for processing by a backend (e.g., an Auto Scaling Group of EC2 instances or Lambda functions). This smooths out "spiky" traffic, preventing the backend from being overwhelmed and allowing it to process items at its own pace.

Scalability capabilities

Amazon EC2 Auto Scaling: The service that performs horizontal scaling for EC2.

  • Auto Scaling Group (ASG): A collection of EC2 instances, defined by a Launch Template (which specifies the AMI, instance type, etc.).
  • Scaling Policies: The rules that trigger scaling.
    • Target Tracking: "Keep average CPU utilization at 50%." (Simplest, most common).
    • Step/Simple Scaling: "If CPU > 80% for 5 mins, add 2 instances."
    • Scheduled Scaling: "Set desired instances to 10 at 9 AM."
  • AWS Auto Scaling: A higher-level service that can automatically scale *other* services, not just EC2 (e.g., DynamoDB capacity, ECS tasks).

Skills in:

Decoupling workloads so that components can scale independently

This is the SQS pattern. Imagine a monolith application that ingests an image, creates a thumbnail, and adds a watermark. This is slow. A high-performance, decoupled design would be:

API Gateway -> Lambda (ingest) -> S3 (raw image) & SQS (job) -> EC2 ASG (thumbnail) -> SQS (job) -> EC2 ASG (watermark) -> S3 (final image).

In this model, the "thumbnail" ASG can scale independently of the "watermark" ASG based on the depth of its own SQS queue.

Identifying metrics and conditions to perform scaling actions

The metric you scale on must match the bottleneck.

  • CPU-bound application: Scale on Average CPU Utilization.
  • Memory-bound application: Scale on Average Memory Utilization (requires the CloudWatch Agent).
  • Request-bound application: Scale on RequestCountPerTarget (an ALB metric).
  • Queue-processing application: Scale on ApproximateNumberOfMessagesVisible (an SQS metric).

Selecting the appropriate compute options and features

EC2 Instance Families:

  • T-family: (e.g., t3.medium) Burstable. Good for low-to-moderate baseline CPU with occasional spikes (e.g., dev/test, small websites).
  • M-family: (e.g., m5.large) General Purpose. A balance of CPU, memory, and network. The workhorse.
  • C-family: (e.g., c5.xlarge) Compute Optimized. High CPU. Good for media encoding, batch processing, CPU-bound web servers.
  • R-family: (e.g., r5.xlarge) Memory Optimized. High RAM. Good for in-memory caches (ElastiCache), large databases (RDS), or memory-intensive analytics.

EC2 Placement Groups: A logical grouping of instances to influence their placement for performance.

  • Cluster: Packs instances close together in a single AZ. Use for lowest latency and highest network bandwidth (e.g., High-Performance Computing).
  • Spread: Spreads instances across different underlying hardware (and different racks). Use for high availability and to reduce correlated failures (e.g., for critical databases or a small number of instances).
  • Partition: Spreads instances across different "partitions" (groups of racks) within an AZ. Use for large, distributed workloads like HDFS, HBase, Cassandra.

EC2 Enhanced Networking: Uses Single Root I/O Virtualization (SR-IOV) for high-performance networking. Provides higher bandwidth, higher Packets Per Second (PPS), and lower latency. It's enabled by default on all modern instance types.

Selecting the appropriate resource type and size

For AWS Lambda, performance is directly tied to the Memory setting. When you increase a function's memory, you also proportionally increase its CPU and network performance. If your function is slow, increasing its memory (e.g., from 128MB to 512MB) is the primary way to make it run faster.

3.3: Determine high-performing database solutions

This section is critical. It covers choosing the right database service for your workload's access patterns and performance needs.

Knowledge of:

Database types and services (relational, non-relational, in-memory)

  • Relational (OLTP): Amazon RDS. Your classic SQL database (MySQL, PostgreSQL, SQL Server, etc.). Used for structured data, complex queries, and transactions (e.g., e-commerce, financial systems).
    • Amazon Aurora: AWS's "cloud-native" relational database. It's compatible with MySQL and PostgreSQL but has much higher performance, durability (data replicated 6 ways across 3 AZs), and faster failover. This is often the correct "high-performance" RDS answer.
    • Aurora Serverless: An on-demand, auto-scaling version of Aurora. Perfect for intermittent or "spiky" database workloads.
  • Non-relational (NoSQL): Amazon DynamoDB. A fully managed key-value and document database. Delivers consistent, single-digit millisecond latency at *any* scale. Use for high-traffic web apps, mobile apps, gaming, IoT—anywhere you need high speed and massive, predictable scale, but don't need complex joins.
  • In-Memory: Amazon ElastiCache. A managed cache service (Redis or Memcached). Used to store temporary data in RAM for sub-millisecond access.

Data access patterns (read-intensive vs. write-intensive)

This is key to your design.

  • Read-Intensive Workload: (e.g., a blog, a product catalog). The best way to scale is with Read Replicas (for RDS/Aurora) and Caching (ElastiCache, DynamoDB DAX).
  • Write-Intensive Workload: (e.g., IoT data ingestion, logging). This is harder to scale. For RDS, you must Vertical Scale (a bigger instance) or "shard" (split) your data. For DynamoDB, you just increase the Write Capacity Units (WCUs). DynamoDB is built for write-heavy workloads.

Database capacity planning

  • RDS: You provision an instance size (e.g., db.r5.large) and EBS volume (size and Provisioned IOPS). You can scale these up.
  • DynamoDB: You provision Read Capacity Units (RCUs) and Write Capacity Units (WCUs).
    • Provisioned Mode: You set the RCU/WCU manually. Good for predictable workloads.
    • On-Demand Mode: DynamoDB instantly scales to meet traffic. Pay-per-request. Good for unpredictable or "spiky" workloads.
Exam Tip: For DynamoDB, use Provisioned Mode with Auto Scaling for predictable traffic patterns. Use On-Demand Mode for new applications, spiky traffic, or when you can't predict capacity. On-Demand is often simpler and can be cheaper than heavily overprovisioning.

Database connections and proxies

Amazon RDS Proxy: A managed database connection pooler that sits in front of RDS. It's crucial for performance in serverless (Lambda) applications, as it pools and reuses database connections, avoiding the high cost of opening/closing new connections for every function invocation. It also improves failover times.

Database engines and migrations

AWS Database Migration Service (DMS): A service to help you migrate databases to AWS.

  • Homogeneous Migration: Same engine to same engine (e.g., on-premises MySQL to RDS MySQL).
  • Heterogeneous Migration: Different engine to different engine (e.g., on-premises Oracle to RDS PostgreSQL). This also requires the Schema Conversion Tool (SCT).
  • Determining the right engine (e.g., MySQL vs. PostgreSQL) depends on application compatibility, cost, and feature requirements.

Database replication (for example, read replicas)

This is the most important concept for scaling database reads.

  • RDS Read Replicas:
    • Used for read scaling.
    • Use asynchronous replication. (This means there can be a small "replication lag".)
    • Can be in the same AZ, a different AZ, or a different Region.
    • You can have up to 15 replicas (for MySQL/PostgreSQL).
  • RDS Multi-AZ:
    • Used for High Availability (HA) / failover.
    • Uses synchronous replication to a standby in a different AZ. (No data loss).
    • You *cannot* send read traffic to the standby. It's just for failover.
  • Amazon Aurora: Aurora's architecture is different. Its single storage volume is shared across 3 AZs. Aurora Replicas (up to 15) all share this volume, have sub-10ms replication lag, and also act as HA failover targets. They are superior to standard RDS Read Replicas.
Exam Tip: HA (Failover) = RDS Multi-AZ. Performance (Scaling) = Read Replicas. Aurora Replicas do both.

AWS global infrastructure

Amazon Aurora Global Database: A high-performance option for global applications. It consists of one primary region (for reads/writes) and one or more secondary regions (read-only). It uses dedicated infrastructure for replication with sub-second latency, allowing for fast global reads and a low-RTO (sub-minute) DR failover to a secondary region.

Skills in:

Designing database architectures

This skill involves combining the knowledge above. For example, designing a global, read-heavy application would involve Aurora Global Database. Designing a serverless API backend would involve DynamoDB or Aurora Serverless with RDS Proxy.

Configuring read replicas to meet business requirements

If an application is read-heavy and the primary database is overloaded, the solution is to create one or more Read Replicas and direct the application's read traffic (e.g., SELECT statements) to the replica endpoint.

Determining an appropriate database type (e.g., Aurora vs. DynamoDB)

This is a classic exam question.

  • Need complex SQL joins, transactions, and structured data? RDS or Aurora.
  • Need massive scale, millisecond latency, and have simple key-value lookups? DynamoDB.
  • Migrating a "lift-and-shift" enterprise app (Oracle, SQL Server)? RDS.
  • Building a new, high-performance cloud-native relational app? Aurora.

Integrating caching to meet business requirements

If an RDS-based application is slow due to complex, frequent read queries, ElastiCache is the solution. If a DynamoDB-based application is read-heavy (or has a "hot key"), DAX is the solution.

3.4: Determine high-performing and/or scalable network architectures

This section is about making your network fast, scalable, and resilient, especially for global or hybrid applications.

Knowledge of:

Edge networking services

These services accelerate content delivery to users by using AWS's global edge network.

  • Amazon CloudFront: A Content Delivery Network (CDN). It caches your static (images, video, JS) and dynamic content at Edge Locations around the world, close to your users. When a user requests content, it's served from the nearest edge location, which is much faster than fetching it from your origin (e.g., S3 or ALB) in a single region.
  • AWS Global Accelerator: This service improves the performance and availability of your *applications* for a global audience. It uses the AWS global network to route user traffic to the optimal application endpoint (e.g., the closest, healthiest Region). It gives you two static Anycast IP addresses that act as a fixed entry point to your application, simplifying failover and firewall rules.
Exam Tip: CloudFront vs. Global Accelerator
  • Use CloudFront to cache *content* (images, videos, static files) at the edge for fast delivery. It's for HTTP/S.
  • Use Global Accelerator to get a static IP and route *application traffic* (HTTP, TCP, UDP) over the fast AWS global network to the best endpoint. It's for non-cachable dynamic content, gaming, IoT, or VoIP.

How to design network architecture (e.g., subnet tiers, routing, IP addressing)

This refers to the multi-tier VPC design (public/private subnets) and the routing tables that control traffic. For performance, a key concept is VPC Endpoints (covered in 1.2). By using Gateway or Interface Endpoints, you keep traffic destined for AWS services (like S3 or DynamoDB) *within* the high-performance AWS network, rather than sending it over the internet.

Load balancing concepts

This is a recap of 2.1, but with a performance focus.

  • Application Load Balancer (ALB): Layer 7. Flexible, great for web apps.
  • Network Load Balancer (NLB): Layer 4. Highest performance. Can handle millions of requests per second at ultra-low latency. It preserves the source IP address of the client. Ideal for high-performance TCP/UDP applications.

Network connection options

This is about connecting your on-premises data center to AWS.

  • AWS VPN: An encrypted IPsec tunnel over the public internet. Performance can be variable.
  • AWS Direct Connect (DX): A dedicated private physical connection (1 Gbps, 10 Gbps, 100 Gbps). Bypasses the internet. Provides consistent, high-bandwidth, low-latency performance. This is the high-performance solution for hybrid connectivity.
  • AWS PrivateLink: This is the technology behind Interface Endpoints. It allows you to *privately* access services (even third-party SaaS services) from your VPC as if they were running locally, without using the internet, NAT Gateways, or VPN.

Skills in:

Creating a network topology for various architectures

  • Global: Use Route 53 for DNS, CloudFront for cached content, and Global Accelerator for application endpoints. Deploy your application in multiple Regions.
  • Hybrid: Use Direct Connect for high-performance, consistent connectivity between on-premises and your VPC. Use a Transit Gateway to act as a central hub to connect multiple VPCs and your on-premises network.
  • Multi-tier: Use public/private subnets, ALBs, and Security Groups to segment traffic.

Determining network configurations that can scale to accommodate future needs

A key scalable design is using a Transit Gateway (TGW). In a growing organization, instead of "meshing" every new VPC to every other VPC with VPC Peering (which becomes complex), you connect all VPCs to a central TGW. The TGW handles all the inter-VPC and on-premises routing, acting as a scalable cloud router.

Determining the appropriate placement of resources

For high performance:

  • Place EC2 instances that need to communicate with each other in a Cluster Placement Group. This packs them into a single AZ to get the lowest latency and highest network bandwidth.
  • Place your database (RDS) in the same Region as your application (EC2/Lambda) to minimize latency.
  • For global users, place your content on CloudFront.

Selecting the appropriate load balancing strategy

  • Scenario: A standard web application with microservices. Answer: Application Load Balancer (ALB).
  • Scenario: A high-performance gaming application running on UDP. Answer: Network Load Balancer (NLB).
  • Scenario: You need a static IP for your load balancer that is used by clients. Answer: Network Load Balancer (NLB).

3.5: Determine high-performing data ingestion and transformation solutions

This section is about getting data *into* AWS (ingestion) and changing its format (transformation) efficiently.

Knowledge of:

Data ingestion patterns (for example, frequency)

  • Streaming: Continuous, real-time data (e.g., IoT sensor data, clickstreams, logs).
  • Batch: Large, discrete chunks of data (e.g., daily sales reports, periodic backups).

Data transfer services

  • AWS DataSync: A high-performance service for online batch data transfer. It accelerates moving large amounts of data between on-premises (NFS, SMB) and AWS storage (S3, EFS, FSx). It's much faster than using open-source tools over the internet.
  • AWS Storage Gateway: (Covered in 3.1). A hybrid solution for *ongoing* use of cloud storage from on-premises, not a one-time migration.
  • AWS Transfer Family: A fully managed service for file transfers *into and out of* Amazon S3 using protocols like SFTP, FTPS, and FTP. Use this when you need to securely exchange files with third parties.
  • AWS Snow Family (Snowball Edge, Snowmobile): Physical devices used to transfer petabyte-scale data when network transfer is not feasible.

Streaming data services

Amazon Kinesis: The platform for real-time streaming data.

  • Kinesis Data Streams: The core service. A "stream" is made of one or more "shards." You ingest data into shards, and consumers (like Lambda or Kinesis Data Analytics) process it in real-time. You manage the number of shards to scale throughput.
  • Kinesis Data Firehose: The *easiest* way to ingest streaming data. It's fully managed. You just point a source (like Data Streams) at it, and Firehose automatically batches, (optionally) transforms, and loads the data into a destination like S3, Redshift, or Splunk.
Exam Tip: Kinesis Data Streams vs. Firehose
  • Use Data Streams when you need *real-time processing* (sub-second) or multiple, custom applications consuming the same stream. You manage scaling (shards).
  • Use Data Firehose when your goal is simply to *load* streaming data into a destination (like S3 or Redshift). It's fully managed and includes batching and transformation.

Data transformation services

AWS Glue: A fully managed ETL (Extract, Transform, Load) service. You point it at your data (e.g., in S3), and the Glue Data Catalog crawls it to discover the schema. You can then use Glue to run serverless Spark jobs to transform the data (e.g., convert CSV to Parquet) and load it into a data warehouse.

Data analytics and visualization services

  • Amazon Athena: A serverless, interactive query service. It allows you to run standard SQL queries directly on your data in S3 (e.g., query your log files). You only pay for the queries you run.
  • AWS Lake Formation: A service that simplifies setting up a secure data lake in S3. It helps manage data ingestion, cataloging, and provides fine-grained (table and column-level) access control.
  • Amazon QuickSight: A serverless Business Intelligence (BI) service. It's used to create interactive dashboards and visualizations from your data.

Secure access to ingestion access points

This involves using services like VPC Endpoints to allow services (like Kinesis) to be accessed privately from your VPC. It also involves using IAM roles and policies to grant services (like Kinesis Firehose) permission to write to destinations (like S3).

Skills in:

Designing data streaming architectures

A common high-performance pattern: IoT Devices -> Kinesis Data Streams (ingestion) -> Kinesis Data Analytics (real-time analysis) -> Kinesis Data Firehose (batching & loading) -> S3 (data lake storage) -> Athena (ad-hoc SQL querying).

Designing data transfer solutions

  • Requirement: One-time, 100TB migration of a file share from on-premises to S3 over a 1 Gbps link. Solution: AWS DataSync.
  • Requirement: Securely receive daily CSV files from a partner via SFTP. Solution: AWS Transfer Family.
  • Requirement: A 500 PB (petabyte) data transfer from a data center with no internet. Solution: AWS Snowmobile (a literal truck). For <10 PB, use AWS Snowball Edge.

Building and securing data lakes

This involves using AWS Lake Formation to define security policies and access controls for your S3 data lake, and using AWS Glue to catalog the data so it can be queried by Amazon Athena.

Selecting appropriate compute options for data processing

This involves choosing the right tool for transformation. AWS Glue is serverless and best for many ETL jobs. For large-scale big data processing (Spark, Hadoop), Amazon EMR (Elastic MapReduce) provides a managed cluster, offering more control and power.

Implementing visualization strategies

This skill involves using Amazon QuickSight to connect to your data sources (like Athena, RDS, or S3) to build and share interactive BI dashboards with business users.

Transforming data between formats (for example, .csv to .parquet)

Parquet is an open-source, columnar storage format. Storing data in a columnar format (instead of row-based like CSV) is *dramatically* more performant and cost-effective for analytics queries (like those from Athena), because the query only needs to read the columns it cares about.

The standard tool to perform this CSV-to-Parquet transformation is AWS Glue.

Exam Tip: Any time you see a question about optimizing Athena query performance or cost, the answer is almost always to convert the data to Parquet and/or partition the data (e.g., by date).

4.1: Design cost-optimized storage solutions

This entire domain is about saving money. This section focuses on choosing the right storage service, tier, and lifecycle policy to meet requirements at the lowest possible cost.

Knowledge of:

AWS cost management tools

  • AWS Cost Explorer: A tool to visualize and forecast your costs and usage over time. You can filter and group data (e.g., by service, by tag) to understand your spending.
  • AWS Budgets: Allows you to set custom budgets (e.g., "$500/month for EC2") and receive alerts when your costs or usage *exceed* (or are forecasted to exceed) your budget.
  • Cost Allocation Tags: These are labels (key-value pairs) you apply to your resources (like EC2, S3, RDS). You can "activate" these tags in the billing console to see a detailed cost breakdown by tag (e.g., cost by "Project", "Department", or "Environment").
  • AWS Cost and Usage Report (CUR): The most detailed billing report. It's a large CSV file delivered to S3 that contains line-item level detail of all your charges.
  • Multi-Account Billing: With AWS Organizations, all costs roll up to the management account, allowing you to see a consolidated bill.

Storage types and AWS storage services

The key here is that different services have different cost models:

  • S3 (Object): You pay for storage (per GB/month) AND data transfer (per GB) AND requests (per 1,000 requests). It's very cheap for storage. S3 also has an access option called Requester Pays, where the *requester* (not the bucket owner) pays for data transfer costs.
  • EBS (Block): You pay for the storage you *provision* (per GB/month), regardless of whether you use it. You also pay for IOPS (for io1/io2/gp3).
  • EFS (File): You pay for the storage you *use* (per GB/month). It's more expensive than S3 or EBS, but you're paying for the convenience of an elastic, shared file system.
  • Amazon FSx: Managed file systems (for Windows File Server, Lustre). More specialized and priced based on provisioned throughput, storage, and other features. FSx for Windows is the cost-optimized choice for "lift-and-shift" Windows applications.

Storage tiering (for example, cold tiering for object storage)

This is the most important topic for S3 cost optimization. You move data to cheaper tiers as it becomes less frequently accessed.

  • S3 Standard: For frequent access. Most expensive, highest performance.
  • S3 Standard-Infrequent Access (S3-IA): For long-lived, less-frequently accessed data. Cheaper storage price, but has a *per-GB retrieval fee*. (Min. 30 days).
  • S3 One Zone-IA: Same as S3-IA, but stores data in only *one* AZ. It's 20% cheaper. Use for data that is non-critical and can be easily recreated.
  • S3 Glacier Instant Retrieval: For long-term archival with *millisecond* retrieval. Cheaper storage than S3-IA, but higher retrieval fee. (Min. 90 days).
  • S3 Glacier Flexible Retrieval: (Formerly S3 Glacier). For long-term archival, retrieval takes *minutes to hours*. Very cheap storage. (Min. 90 days).
  • S3 Glacier Deep Archive: The cheapest storage on AWS. For long-term archival where retrieval time of *12+ hours* is acceptable. (Min. 180 days).
Exam Tip: For unknown or changing access patterns, S3 Intelligent-Tiering is the best choice. It monitors your data's access patterns and automatically moves objects to the most cost-effective tier (e.g., from frequent to infrequent access) without any performance impact or retrieval fees.

Data lifecycles & Storage access patterns

You must understand the access pattern to choose the right tier.

  • Pattern: User-uploaded photos, accessed daily. Tier: S3 Standard.
  • Pattern: Monthly log files, accessed once a week for the first month, then rarely. Tier: Start in S3 Standard, use a Lifecycle Policy to move to S3-IA or Glacier.
  • Pattern: 10 years of compliance data that must be kept but will likely never be accessed. Tier: S3 Glacier Deep Archive.

Block storage options (HDD vs. SSD)

For EBS, you can choose SSD (fast, expensive) or HDD (slow, cheap).

  • General Purpose SSD (gp3): Best for most workloads (boot volumes, dev/test).
  • Provisioned IOPS SSD (io2): Best for high-performance databases.
  • Throughput Optimized HDD (st1): Low-cost magnetic storage. Good for *high-throughput*, large, sequential I/O (e.g., data warehouses, log processing).
  • Cold HDD (sc1): The cheapest EBS volume. For *infrequently* accessed data.

Backup strategies & Hybrid storage options

Backup Strategies: Don't pay for backups you don't need. Configure your AWS Backup retention policies to match your business requirements, not longer.

Hybrid Storage: Using Storage Gateway can be a cost-optimization strategy. For example, using S3 File Gateway allows you to store the bulk of your data in low-cost S3 while keeping only a small, frequently-accessed cache on-premises, reducing your on-premises storage footprint.

Skills in:

Managing S3 object lifecycles

This is the skill of creating the automation rules (Lifecycle Policies) to move data between storage tiers or delete it, based on the object's age.

Selecting the most cost-effective storage service for a workload

  • Workload: Backups. Cost-effective: S3, with a lifecycle policy to Glacier Deep Archive.
  • Workload: Shared file system for EC2. Cost-effective: EFS. (Don't try to build your own NFS server on EC2+EBS, it's not as cost-effective or resilient).
  • Workload: Boot volume. Cost-effective: EBS gp3.
  • Workload: Big data log processing. Cost-effective: EBS st1 or S3.

Designing appropriate storage strategies

This includes choices like S3 upload strategy. For large files, using S3 Multipart Upload is more resilient and performant. For ingesting many small files, it may be more cost-effective (and performant) to batch them into larger archives (like .zip or .tar.gz) before uploading, as this reduces the per-request cost.

Determining storage size and auto scaling

This skill is about "right-sizing." For EBS, this means provisioning a volume that is large enough, but not *too* large. For EFS and S3, scaling is automatic. For FSx, you must provision a specific size, so it's important to monitor and resize as needed.

Selecting backup, archival, and data migration solutions

This is about choosing the lowest-cost solution.

  • Backup/Archival: Use AWS Backup to manage backups and tier them to cold storage. Use S3 Glacier Deep Archive for long-term archival.
  • Migration: For online migration, DataSync is often the most cost-effective, high-speed option. For offline, use Snowball Edge.

Real-World Example: Secure Data Archival

Scenario: Secure, Cost-Effective Data Archival

A financial services company has 50 TB of end-of-year trade data. Compliance rules state this data must be kept for 7 years. It must be 100% secure and un-changeable. It will *probably* never be accessed, but if a regulator asks for it, they must be able to retrieve it within 48 hours.

The "Client" (Business) Perspective:

  • "I have to keep this data for 7 years, or I'll be fined." (Need: Durability, Retention Policy)
  • "Nobody is allowed to delete or change this data. Not even our own admins." (Need: WORM / Immutability)
  • "I'm 99% sure we'll never look at it. I want to pay the *absolute minimum* cost for this." (Need: Cost-Optimization)
  • "If we do need it, getting it back in 2 days is fine." (Need: Non-urgent Retrieval Time)

The "Expert" (Architect) Solution:

This is a perfect use case for S3 storage classes and lifecycle policies.

  1. Storage Service (Domain 4): We will use S3 Glacier Deep Archive. This is the cheapest storage on AWS, designed for exactly this "write-once, access-never" pattern.
  2. Immutability (Domain 1): To meet the "no-deletes" requirement, we will enable S3 Object Lock on the bucket in "Compliance Mode." This makes the data immutable for the 7-year retention period. *Nobody* (not even the root user) can delete the objects.
  3. Automation (Domain 4): We will create an S3 Lifecycle Policy. All data will first be uploaded to an S3 "landing" bucket (in S3 Standard). The lifecycle policy will automatically transition all objects to S3 Glacier Deep Archive after 1 day.
  4. Retrieval (Domain 2): The client's 48-hour (2-day) retrieval RTO is easily met. S3 Glacier Deep Archive has a standard retrieval time of 12 hours.

Why this is the perfect solution:

  • Cost (Domain 4): This is the primary driver. By using S3 Glacier Deep Archive, the client will pay a tiny fraction of the cost of storing this data in S3 Standard.
  • Compliance (Domain 1): S3 Object Lock meets the strict "immutable" compliance requirement.

Key Services Used: S3 Standard, S3 Lifecycle Policy, S3 Glacier Deep Archive, S3 Object Lock.

4.2: Design cost-optimized compute solutions

Compute is often the largest part of an AWS bill. This section is about reducing that cost through purchasing options and right-sizing.

Knowledge of:

AWS cost management tools

Use Cost Explorer to analyze compute spend, Cost Allocation Tags to assign costs to projects/teams, and AWS Budgets to get alerted if compute costs are rising unexpectedly.

AWS purchasing options

This is the single most important topic for compute cost-optimization.

  • On-Demand: Pay-by-the-hour/second. No commitment. Most expensive, most flexible. Use for unpredictable workloads, or dev/test.
  • Reserved Instances (RIs): A 1-year or 3-year commitment to a specific instance family, region, (and sometimes AZ). Gives a large discount (up to 72%) vs. On-Demand. Use for *steady-state*, predictable workloads (e.g., your production database or web servers that run 24/7).
  • Savings Plans: A 1-year or 3-year commitment to a specific *dollar amount* of compute spend (e.g., "$10/hour"). This is more flexible than RIs.
    • Compute Savings Plan: Applies to any EC2 instance family, in any region, and also to Fargate and Lambda. Most flexible.
    • EC2 Instance Savings Plan: Applies to a specific instance family in a specific region (like an RI), but not to a specific AZ.
  • Spot Instances: Uses spare, unused EC2 capacity at a *massive* discount (up to 90%). The catch: AWS can terminate your instance with a 2-minute warning. Use for: Fault-tolerant, stateless workloads that can be interrupted. Examples: batch processing, data analysis, CI/CD jobs. Do *not* use for databases or critical servers.
Exam Tip: Savings Plans vs. Reserved Instances
  • Savings Plans (Compute) are the new, more flexible standard. They apply to EC2, Fargate, and Lambda. They are the best choice for most general-purpose cost savings.
  • Reserved Instances (Standard) are still useful if you need to guarantee capacity in a *specific Availability Zone*. Convertible RIs offer less discount but let you change instance families.

Instance types, families, and sizes (for example, memory optimized, compute optimized, virtualization)

This is about Right-Sizing. Don't use a c5.2xlarge (8 vCPUs) for a small web server that only uses 5% CPU. Use a smaller, cheaper instance (like a t3.medium). Use tools like AWS Compute Optimizer to get recommendations on right-sizing your instances based on their actual CloudWatch metrics.

Optimization of compute utilization

The most cost-optimized compute is compute you don't pay for.

  • Serverless (Lambda, Fargate): You pay *only* for what you use, down to the millisecond. You don't pay for idle servers. This is extremely cost-effective for "spiky" or unpredictable traffic.
  • Containers (ECS/Fargate): Containers allow you to "bin-pack" multiple applications onto a single EC2 instance, improving its utilization.
  • Auto Scaling: By scaling in (terminating instances) when traffic is low, you stop paying for compute you don't need.

Scaling strategies (for example, auto scaling, hibernation)

EC2 Hibernation: For instances that are used intermittently (e.g., a dev desktop used 9-5). When "stopped", the instance's RAM is saved to the EBS volume. When "started", it resumes from its previous state. This gives you a faster startup time, and you only pay for the EBS storage while it's hibernated (not the instance cost).

Distributed and hybrid compute

  • Distributed Compute: Using edge services like CloudFront and Lambda@Edge can reduce compute costs by caching responses at the edge, reducing the load on your origin compute.
  • Hybrid Compute: AWS Outposts brings AWS racks to your data center, but this is a high-cost solution for low-latency needs. AWS Snowball Edge can provide compute capabilities in disconnected environments.

Skills in:

Determining cost-effective AWS compute services

  • Workload: Unpredictable API, runs 10 times a second. Cost-effective: Lambda.
  • Workload: 24/7 production web server. Cost-effective: EC2 with Savings Plans or Reserved Instances.
  • Workload: Batch data processing job that runs nightly and can be interrupted. Cost-effective: Spot Instances.
  • Workload: Migrating a containerized application. Cost-effective: Fargate (to avoid managing/paying for idle EC2 capacity).

Determining an appropriate load balancing strategy

While ALB/NLB/GWLB have different performance, they also have different costs. An ALB or NLB is generally low-cost. A GWLB is for specialized (and often costly) third-party appliances. Don't provision a load balancer if it's not needed (e.g., for a single-instance dev server).

Determining availability for workloads

Don't pay for high availability you don't need. A production workload needs a Multi-AZ architecture. A non-production or dev/test workload can run on a single instance in a single AZ, which is significantly cheaper.

Selecting the appropriate instance family for a workload

This is about right-sizing. Don't use a Memory Optimized (R-family) instance for a CPU-bound (C-family) job. Use the instance family that matches your bottleneck. AWS Compute Optimizer is the tool for this.

4.3: Design cost-optimized database solutions

This section focuses on choosing the right database service, capacity mode, and data policies to minimize cost.

Knowledge of:

AWS cost management tools

Use Cost Explorer, AWS Budgets, and Cost Allocation Tags to track your database spend. Use the AWS Cost and Usage Report (CUR) for highly detailed analysis. In a multi-account setup, all database costs will roll up to the management account.

Database types and services (e.g., relational vs. non-relational, Aurora, DynamoDB)

Choosing the wrong database is a common cost mistake.

  • Don't use RDS/Aurora for simple key-value lookups. DynamoDB is often more cost-effective at scale for this pattern.
  • Don't use DynamoDB if you need complex SQL joins. The application-side logic to "fake" joins will be complex and expensive.

Database capacity planning (for example, capacity units)

This is a major cost optimization skill.

  • RDS: Change the instance size (e.g., db.r5.large -> db.r5.medium) if it's overprovisioned.
  • DynamoDB:
    • Provisioned Mode: Use Auto Scaling to scale your RCUs/WCUs up and down with traffic. Don't leave them provisioned for peak 24/7.
    • On-Demand Mode: Use this for new or unpredictable workloads. It's often cheaper than overprovisioning in Provisioned Mode.

Caching strategies

Using a cache like Amazon ElastiCache can reduce cost by offloading read traffic from your database, allowing you to run a smaller, cheaper database instance.

Amazon Aurora Serverless

Amazon Aurora Serverless: An on-demand, auto-scaling version of Aurora. It's perfect for intermittent, "spiky", or unpredictable database workloads (like dev/test). It can scale *down to zero*, at which point you pay only for storage. This is a huge cost-saver compared to running a traditional RDS instance 24/7.

Database connections, proxies, engines, and migrations

  • Connections/Proxies: Using RDS Proxy can save costs with Lambda by reducing the number of connections and allowing for more efficient processing.
  • Engines/Migrations: Migrating from a commercial engine (like Oracle or SQL Server) to an open-source engine (like PostgreSQL or MySQL) using DMS and SCT can dramatically reduce licensing costs.

Database replication and data retention

  • Replication: Read Replicas add cost (for the instance and storage). Only use them if you have a clear performance need.
  • Retention: Don't store data you don't need. For RDS, automated backups and snapshots cost money. Configure your retention policy (e.g., "keep backups for 7 days, not 30") to match your business requirements.

Skills in:

Designing appropriate backup and retention policies

If you have an automated RDS backup with a 7-day retention, but you also have a manual snapshot that is 3 years old, investigate if that snapshot is still needed. It's costing you money every month in S3 storage.

Determining cost-effective AWS database services

  • Workload: Dev/Test database, used 9-5 on weekdays. Cost-effective: Aurora Serverless, which can scale to 0 when not in use.
  • Workload: Massive-scale IoT data ingestion. Cost-effective: DynamoDB (not RDS, which would require a huge, expensive instance).
  • Workload: 24/7 production database. Cost-effective: RDS or Aurora, paid for with Reserved Instances.

Determining cost-effective database engines and types

  • Engine: To save on licensing, migrate from RDS for Oracle to RDS for PostgreSQL.
  • Type (Time Series, Columnar): For analytics queries, storing data in a columnar format (e.g., in Redshift or as Parquet in S3) is far more cost-effective than querying an RDS instance. For time-series data (like IoT metrics), Amazon Timestream may be more cost-effective than DynamoDB.

Migrating database schemas and data

This skill involves using AWS DMS and SCT to perform heterogeneous migrations, with the primary goal of moving to a more cost-effective, open-source, or cloud-native database engine.

4.4: Design cost-optimized network architectures

Network costs are often overlooked. The primary cost is Data Transfer Out (DTO) to the internet. Data transfer *into* AWS is free, and data transfer *within* a Region is often free or cheap.

Knowledge of:

AWS cost management tools

Use Cost Explorer to analyze DTO costs. Use Cost Allocation Tags on resources like NAT Gateways to track which projects are generating network traffic.

Load balancing concepts

Load Balancers (ALB, NLB) have both a fixed hourly cost and a data processing fee. Ensure you are using them efficiently and not leaving idle load balancers running in test accounts.

NAT gateways (for example, NAT instance costs compared with NAT gateway costs)

Instances in a private subnet need a way to access the internet.

  • NAT Gateway: Fully managed, scales automatically, highly available. You pay a per-hour fee *and* a per-GB data processing fee. This can get expensive.
  • NAT Instance: An EC2 instance (e.g., t3.micro) in a public subnet that you configure (with a script) to do routing. Cost: You just pay for the EC2 instance (which can be a cheap burstable instance or even a Spot instance). Downside: You must manage it, scale it, and it's a single point of failure (unless you build an HA architecture for it).
Exam Tip: For *low-traffic* environments (like dev/test), a NAT Instance is often more cost-effective. For *high-traffic*, critical production workloads, the reliability of a NAT Gateway is worth the cost.

Network connectivity (for example, private lines, dedicated lines, VPNs)

  • AWS Direct Connect (DX): High bandwidth, but has a fixed monthly port fee. The *data transfer out* (DTO) rate over DX is significantly *cheaper* than DTO over the internet.
  • AWS VPN: Cheaper setup (no port fee), but data transfer costs are the same as standard internet DTO.

Skill: If you are transferring *massive* amounts of data (terabytes) out of AWS back to your on-premises data center every month, Direct Connect will be more cost-effective than VPN, despite its higher setup cost.

Network routing, topology, and peering (for example, AWS Transit Gateway, VPC peering)

When connecting VPCs, VPC Peering is just a network connection. Data transfer across peering connections *within the same region* is free.

AWS Transit Gateway: Has a per-hour cost and a per-GB data processing fee. This is simpler to manage at scale, but you must account for the TGW processing fee in your costs.

Network services (for example, DNS)

Amazon Route 53 has costs for hosted zones and for queries. For internal DNS, you can use VPC Route 53 Resolver, which has different cost implications.

Configuring appropriate network routes to minimize network transfer costs

This is the most important skill in this section.

  • Problem: Your private EC2 instance is talking to S3, and the traffic is going through your NAT Gateway. Cost: You are paying the NAT Gateway data processing fee for every GB. Solution: Create a Gateway VPC Endpoint for S3. Route traffic to S3 through this endpoint. The endpoint is *free* and the data transfer is *free*, as it stays within the AWS network.
  • Problem: Your EC2 instances in 3 different AZs are all sending traffic to a single NAT Gateway in AZ-A. Cost: You are paying Inter-AZ data transfer fees for the instances in AZ-B and AZ-C. Solution: Create a NAT Gateway *in each AZ* and configure the route tables so instances use their local NAT Gateway. This eliminates the Inter-AZ cost.
Exam Tip: Remember that data transfer *between Availability Zones* in the same region is *not free*. You are billed for traffic in both directions. This is why placing your NAT Gateway in each AZ (as in the example above) is a key cost-saving and resilient pattern.

Determining strategic needs for content delivery networks (CDNs) and edge caching

Amazon CloudFront (CDN): This is a key cost-optimization tool. When CloudFront serves a cached file from an edge location, it's *cheaper* than serving it from your origin (S3 or ALB), because the DTO rates from CloudFront are lower. It also reduces the load (and cost) on your origin servers.

Reviewing existing workloads for network optimizations

This involves using tools like VPC Flow Logs to analyze traffic patterns. You might discover that a misconfigured application is sending terabytes of data over the internet or across regions, incurring unnecessary costs that could be fixed with a VPC Endpoint or a change in architecture.

Selecting appropriate throttling and bandwidth allocation

  • Throttling: Using API Gateway to throttle requests can prevent runaway costs from a misbehaving client or a malicious attack.
  • Bandwidth: For Direct Connect, you must select the right bandwidth (e.g., 1 Gbps vs. 10 Gbps). Overprovisioning will cost you money, while underprovisioning will create a bottleneck.
